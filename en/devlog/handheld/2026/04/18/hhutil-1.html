hhutil: A toolbox for retro handhelds

# My newest obsession

Recently, I acquired a handheld console, one of those cheap ones, called the R36S. It’s been a few months, actually, but my excitement is so high it feels like I just turned it on for the first time yesterday. The experience reminds me of when I bought my NDS, which I still keep to this day. I could stay here talking about emulators and childhood games, but that’s not what got me excited; it was the memories of “hacking” my DS and installing every kind of Homebrew I could find on the classic GBATemp forums. It was like having a handheld computer before smartphones, and that’s exactly what these retro handhelds are: small ARM-based SBCs running a gaming-focused Linux. And I am completely obsessed.

There are several options for these devices on the market, at all prices and with all sorts of options. The one I have is a clone of a clone; the R36S is just the best-known member of a whole family of clones that basically share the same SoC, the Rockchip RK3326. And it’s a big family; there’s even an entire Wiki dedicated to documenting the different versions and variations appearing out there, as well as helping users install CFWs (Custom Firmware) on their devices. The community is extremely active, just look at the amount of posts made on Reddit every day.

# And for creating games?

These consoles are quite limited, both in terms of hardware and software. According to the Wiki, the standard version of the R36S has the following specs:

Soc RockChip RK3326
CPU Quad-Core ARM Cortex-A35 (1.5 GHz)
GPU Mali-G31 MP2 (650 MHz)
RAM 1GB DDR3L
Display 3.5-inch IPS, 640x480
Wi-Fi External Dongle Only
Bluetooth Still no success

It’s no gaming PC. And in terms of software, it doesn’t seem like a big problem since it runs Linux and should run any language that can be compiled for ARM64. However, this GPU only supports OpenGL ES (which is not the same as OpenGL). Additionally, most of these CFWs do not include Wayland or X11, which makes the process even harder. Luckily, the PortMaster team has been making a great effort to improve compatibility for these devices, but the process is slow and much depends on changes within the engines themselves to work.

This is the case with my favorite engine, Ebitengine. Although in theory it should run without problems, as it supports OpenGL ES by default and Go can be easily compiled for ARM64, my experience wasn’t that simple. The program complained about not finding X11, and I didn’t have much luck with the PortMaster devs. Instead of continuing to bash my head against this problem, I preferred to switch to LÖVE and I’m quite satisfied with that choice. Not only is it a much more robust engine, but it’s also fully compatible with Portmaster without even needing compilation. Just zip the source code into a file with the .love extension and use the script available on their site. So that’s what I did; with some dedication and the help of great documentation, it was easy to do everything I wanted using LÖVE.

# And what did I do?

So, hhutil is a collection of applications created specifically for retro consoles. Using a joystick as a controller and a resolution of only 640x480, these apps seek to deliver a native experience on these devices. I tried to create basic functionalities that make sense on a device like this and don’t require external accessories to function, in most cases. The repository with all applications can be found here, but here’s a brief presentation of each.

# Calc

It’s a calculator with all the functions you expect and then some. In addition to the four basic operations, it also has trigonometric functions, logical operations, different number bases, and works with expressions. I also included a memory system with different variables, making this a quite powerful calculator, even more than some scientific ones on the market. Its algorithm gave me a lot of work and I believe there are still some bugs to be fixed in more complex cases, but overall I was happy with the result.

# Dice

I wanted to make a dice rolling app that had animations but was at the same time minimalist and straight to the point. I think I found a good middle ground with this implementation. I used this 7-segment display aesthetic instead of spinning dice mainly because I wouldn’t need to create any art from scratch, but the result was better than expected. Along with the sound effects, I think it’s a good addition to any game night or TTRPG.

# Metronome

I decided to take a risk by creating a metronome. In terms of design, I liked the result, but I confess my ears aren’t trained to tell if the logic is perfectly on point. I made sure to keep each button with only one function, so you don’t need to look at the screen while using it. I also added a variety of different sounds for variety; I particularly liked the result, and if my musical talents were sharper, I would definitely use this metronome to practice.

# GPTester

This was my favorite one to create. It’s just an app to test the console buttons, but it does more than that. I was inspired by the demos that circulated in the 90s and tried to implement some effects. We have plasma, scrolling text, and shaking text—just the classics. Unfortunately, the GPU creates some artifacts in the shader at times, but the effect ended up looking cool.

# TODO

Now the apps start getting more serious. This is a list where you can mark items as completed; I called it TODO. You can write your shopping list or daily chores. Since we don’t have keyboard inputs, we have a virtual keyboard on the screen. A future improvement could be allowing the user to include items using a computer, but it’s just an idea for now.

# Habit

The first app idea I had was a habit tracker. I’m not sure if I liked the result very much, but what matters is that it’s functional. You can create habits, mark them daily, and track your progress over time. I think it’s one of the most useful ones and the kind of thing that’s cool to have on a device like this that you always carry with you.

# Weather

This is definitely the most complex app of all and the only one that needs the internet to work. It’s just a weather forecast app, but I needed to connect it to some APIs to get this data. The result was great, and I confess I’ve preferred using it over the app on my phone.

# Conclusion

This was just an introduction to the apps I created. I still want to include a calendar app, but I haven’t decided exactly how it should look or what features it will have. It was a good few weeks working on this project, and I learned a lot about how to develop in LÖVE for these retro handhelds; it even made me want to create real games for them. The next step is to trim some remaining edges in the apps and try to include them in the PortMaster catalog. Once that’s done, I want to start bringing some game ideas to these consoles.

I must say I was quite surprised by this engine. I had a prejudice against Lua for being such a simple language, but it’s undeniable how powerful it is: impeccable documentation, built-in physics and networking, real shaders, various community libraries, and even 3D support. It’s no wonder several commercial games use it and it remains relevant even after all this time. The fact that it works so easily with these consoles also pleasantly surprised me. Expect new projects with it in the near future.
